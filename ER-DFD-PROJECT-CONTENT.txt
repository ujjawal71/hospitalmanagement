--- SECTION A: SHORT PROJECT CONTEXT ---
This system lets patients register, log in, book doctor appointments by
department and time slot, and manage their profile. Doctors log in to view
appointments, update status (e.g. PENDING / APPROVED / COMPLETED / CANCELLED),
see patient details, and define time slots. Admins manage departments, doctors,
users, and can view all appointments.

Tech (typical for this repo): Java, Spring Boot, Spring Security, JPA, MySQL;
frontend as static HTML/CSS/JS under src/main/resources/static.

SECTION B: ER DIAGRAM (ENTITY–RELATIONSHIP)
================================================================================

TITLE (for figure above screenshot):
  Figure: Entity–Relationship (ER) Diagram — Doctor Appointment Booking System

DESCRIPTION (paragraph — paste as report text):
The ER diagram shows main data entities and how they connect. ADMIN stores
administrator accounts. DEPARTMENT groups medical areas (Cardiology, Dental,
etc.). PATIENT holds registered users who book visits. DOCTOR belongs to one
department and has specialization and availability-related fields. APPOINTMENT
links one patient and one doctor for a date, time, and status. TIME_SLOT stores
doctor-specific bookable slots (date, start/end time, available flag). Primary
keys (PK) identify rows; unique email (UK) avoids duplicate logins; foreign keys
(FK) tie appointments to patients and doctors, doctors to departments, and
slots to doctors.

ENTITY SUMMARY TABLE (plain text):
  Entity        | PK              | Short description
  -------------|-----------------|------------------------------------------
  Admin        | id              | System administrators
  Department   | id              | Hospital departments
  Patient      | patient_id      | Registered patients
  Doctor       | doctor_id       | Doctors; linked to department
  Appointment  | appointment_id  | Bookings patient–doctor, date/time/status
  TimeSlot     | id              | Doctor availability slots

RELATIONSHIPS (one line each):
  Department 1 ——< many Doctor
  Patient    1 ——< many Appointment
  Doctor     1 ——< many Appointment
  Doctor     1 ——< many TimeSlot

KEY LEGEND:
  PK = Primary Key   UK = Unique   FK = Foreign Key

SECTION C: DFD (DATA FLOW DIAGRAM)
================================================================================

TITLE (Level 0 — above screenshot):
  Figure: DFD Level 0 — Context diagram

TITLE (Level 1 — above screenshot):
  Figure: DFD Level 1 — Processes and data stores

DESCRIPTION — LEVEL 0 (paragraph):
The context diagram shows Patient, Manager (staff), and Admin as external
actors. In this project the actors are Patient, Doctor, and Admin. They all
interact with one central process: “Doctor Appointment Booking System”.
Bidirectional arrows mean data flows both ways (requests from users, responses
and updated data from the system).

DESCRIPTION — LEVEL 1 (paragraph):
Level 1 decomposes the system into processes: Authentication; Manage Doctors
and Departments; Manage Appointments; Manage Time Slots; Manage Users/Profile.
Data stores hold Patients, Doctors, Departments, Appointments, Time Slots, and
Admins. Patients use auth, appointments, and profile processes. Doctors use
auth, appointments, and time slots. Admins use auth, doctor/department
management, appointments, and user/profile management as modeled.

DATA FLOWS SUMMARY TABLE (plain text):
  From    | To     | Data flow (examples)
  --------|--------|--------------------------------------------------
  Patient | System | Login, Register, Book, Cancel, Profile
  Doctor  | System | Login, Appointments, Time slots
  Admin   | System | Login, CRUD doctors/departments, View appointments, Manage users
  System  | Patient| Appointment list, Booking confirmation
  System  | Doctor | Appointment list, Patient details
  System  | Admin  | Lists: doctors, patients, appointments, departments
