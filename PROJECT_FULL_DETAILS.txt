====================================================================
1. PROJECT INTRODUCTION
====================================================================

Ye project ek Hospital Management aur Online Doctor Appointment Booking System hai jisme backend aur frontend dono same codebase ke andar maintain kiye gaye hain. Backend Java Spring Boot par based hai, database MySQL use karta hai, aur frontend static HTML, CSS aur JavaScript files ke through serve hota hai.

Project ka main purpose ye hai:
- Patients ko online appointment booking facility dena
- Doctors ko apne appointments aur time slots manage karne dena
- Admin ko departments, doctors, users aur appointments manage karne dena
- Public visitors ko departments aur doctors ki listing dikhana

Ye application monolithic style me designed hai, matlab backend business logic, security, REST APIs, static frontend, resource configuration aur persistence layer sab ek hi project ke andar hain.

High-level system summary:
- User browser me application open karta hai
- Static frontend pages serve hote hain
- JavaScript browser se REST APIs call karti hai
- Spring Boot controllers requests receive karte hain
- Service layer business logic execute karti hai
- Repository layer MySQL database ke sath interact karti hai
- Response wapas frontend ko milta hai
- SessionStorage me auth token maintain hota hai

====================================================================
2. PROJECT PURPOSE AND BUSINESS PROBLEM
====================================================================

Hospital ya clinic environment me appointment booking traditionally manual hoti hai. Manual system me ye issues hote hain:
- Phone-based booking par dependency
- Doctor availability ko manually track karna padta hai
- Patient records scattered ho sakte hain
- Appointment status updates timely communicate nahi hote
- Admin ke liye doctor aur department management cumbersome ho sakta hai

Ye project in problems ko solve karne ki koshish karta hai by providing:
- Self-service patient registration
- Department-based doctor discovery
- Appointment booking
- Doctor-side approval and status tracking
- Admin-side master data management
- Browser-based role-specific dashboards

Iska functional domain specifically doctor appointment scheduling hai, na ki full enterprise hospital ERP. Matlab billing, pharmacy, laboratory, inventory, bed allocation, insurance, prescriptions aur EMR jaise features is current version me visible nahi hain.

13. AUTHENTICATION AND AUTHORIZATION MODEL
====================================================================

Authentication approach:
- Custom token-based authentication
- Not JWT
- Not session cookie based

Header used:
- X-Auth-Token

Token storage:
- In-memory token store

Token generation:
- UUID-based token generation visible

Authentication flow:
1. User login request send karta hai with email, password, role
2. AuthService repository me matching user search karta hai
3. Password simple equals comparison se verify hota hai
4. TokenStore token generate karta hai
5. Login response me token, role, userId aur name bheja jata hai
6. Frontend sessionStorage me token store karta hai
7. Subsequent API requests me same token header ke through bheja jata hai

Authorization flow:
1. AuthFilter har protected API request intercept karta hai
2. Public paths bypass ho jate hain
3. Missing token par 401 response milta hai
4. Invalid token par 401 response milta hai
5. Wrong role par 403 response milta hai
6. Valid token par userId aur userRole request attribute me set hote hain

Role restriction mapping:
- /api/admin/** => ADMIN
- /api/doctor/** => DOCTOR
- /api/patient/** => PATIENT

Public routes:
- /api/auth/login
- /api/auth/register
- /api/departments/list
- /api/doctors/by-department
- /index.html
- /
- /css/
- /js/
- /pages/

Security design summary:
- Spring Security framework present hai
- But actual authorization largely custom filter ke through implement hai
- SecurityConfig anyRequest().permitAll() use karta hai
- Custom AuthFilter practical guard ka kaam karta hai

====================================================================
14. SECURITY STRENGTHS AND WEAKNESSES
====================================================================

Strengths:
- Role-based route separation hai
- Unauthorized request reject hoti hai
- Token header-based simple auth implement hai
- Session stateless configuration use hoti hai
- Public aur protected paths separated hain

Weaknesses:
- Password hashing visible nahi hai
- Tokens in-memory hain, restart par invalidate ho jayenge
- Multi-instance deployment me token sharing issue hoga
- anyRequest().permitAll() ke karan framework-level authorization strict nahi hai
- Token expiration policy visible nahi hai
- Audit logging visible nahi hai
- Rate limiting visible nahi hai
- Brute-force protection visible nahi hai

Practical implication:
- Ye setup demo, academic ya small local project ke liye okay ho sakta hai
- Production-grade security ke liye significant improvements chahiye honge

- Har API request me X-Auth-Token header automatically inject hota hai
- Role-based redirect dashboards ke liye use hota hai
- requireAuth function unauthorized users ko index.html par bhej deti hai

Advantages:
- Simple to understand
- No extra auth library needed
- Static frontend ke sath compatible

Limitations:
- Browser storage-based token theft risk
- Token expiry handling visible nahi
- Refresh behavior simplistic hai
